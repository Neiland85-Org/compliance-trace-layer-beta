## 3. CONTRACT GAPS (Contratos implícitos)

### 3.1 Deploy Engine API — Sin esquema formal

**POST /deploy** — Contrato implícito:
```
Input:  { name: string, image: string }  ← no validado con schema
Output: { status: "running", container: string, service: string }
Error:  { error: string } | { status: "deploy-failed" }
```
**Problemas:** No hay versión de API, no hay content-type enforcement, no hay schema validation (Zod está instalado pero no se usa).

**POST /deploy-manifest** — Contrato implícito:
```
Input:  { path: string }  ← PATH TRAVERSAL POSIBLE
Output: { status: "stack-started", services: Array }
```
**Problema crítico:** `path` se pasa a `path.resolve()` → cualquier ruta del filesystem es accesible.

### 3.2 Trace API — Sin versionado

**POST /api/trace/create**
```
Input:  { transaction_id: string }  ← validado manualmente (bien)
Output: { transaction_id, hash, status: "verified", timestamp }
```
**Problema:** `status: "verified"` siempre — no hay verificación real.

### 3.3 Frontend → Backend — URL hardcoded

`deployService.js` hace fetch a `http://localhost:4000/deploy` — no usa `VITE_API_URL`.
`apps/console/.env` tiene `VITE_API_URL=http://localhost:5173` — apunta a sí mismo.

### 3.4 Supervisor → Registry — Contrato de estado

El supervisor lee `services.json` y asume que si un container no está en `docker ps`, debe hacer restart. No distingue entre:
- Container detenido intencionalmente
- Container que falló
- Container eliminado por el usuario

**No hay concepto de "desired state" vs "actual state".**

### 4.1 Causas raíz (no síntomas)

| Causa raíz | Manifestaciones | Archivos afectados |

### 4.2 Anti-patterns detectados

1. **God File**: `backend/server.js` aplica helmet, cors y rateLimit dos veces cada uno (líneas 57-63 + líneas 84-89)
2. **Stringly-typed orchestration**: Docker commands construidos como strings concatenados
3. **Optimistic-only error handling**: `catch{}` vacío en múltiples sitios (supervisor, registry reads)
4. **Polling instead of events**: Dashboard.jsx hace polling cada 2s; supervisor cada 5s
5. **Ambient authority**: Deploy engine tiene acceso irrestricto al Docker daemon

## 9. RIESGOS RESIDUALES

1. **Sin autenticación**: Hasta que S-1 se implemente, el deploy engine es accesible sin restricciones desde cualquier origen.
2. **Datos en memoria**: Hasta que S-3 se implemente, todo reinicio del backend pierde todas las trazas.
3. **Docker daemon exposure**: El deploy engine tiene acceso root-equivalent al Docker daemon. Un atacante que llegue al API puede escalar a host.
4. **Sin monitoring**: No hay forma de detectar intrusiones, uso anómalo o degradación de servicio.
